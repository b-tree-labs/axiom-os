# ADR-073 — MCP tool surface is registry-driven (capability = source of truth)

**Status:** Accepted · **Date:** 2026-06-11
**Owner:** @ben
**Builds on:** ADR-056 (skills as invocable functions), ADR-072 + AEOS §4.9 (capability projection), ADR-006 / ADR-038 (MCP access + builtin MCP server)
**Related:** ADR-063 (SKILL.md generated from registry)

## Context

ADR-072 / AEOS §4.9 established that a registered capability (`SkillSpec`) is the single source of truth, and CLI / MCP / SKILL.md / agent-tool are *projections* of it, generated by the shared projector `axiom.infra.capability_projection`. The CLI and agent-tool surfaces already follow this. **MCP does not.**

Today the MCP tool surface is built from a *parallel* manifest convention (`src/axiom/extensions/builtins/mcp/`):

- **Names** are minted in `manifest_schema.py` as `f"{prefix}__{tool_name}"` from `[[extension.mcp.tool]]` blocks — a mangler independent of the projector.
- **Schemas** are `{"type": "object", "additionalProperties": True}` (aggregation.py) or hand-crafted per-tool (`platform_primitives.py`) — not derived from the capability's `inputs`.
- **Handlers** are lazy-imported from each tool's `[[extension.provides kind=tool]].entry = "module:function"` — the registry's `invoke()` is never called.
- Result: **two sources of truth.** A skill gains a parameter and only the CLI/agent-tool sees it; the MCP tool drifts. This is exactly the divergence ADR-072 set out to end.

## Decision

**The MCP tool surface for extension capabilities is derived from the `SkillRegistry` via the shared projector — not from manifest `[[mcp.tool]]` blocks.**

1. **Discovery.** For each registered capability whose manifest declares `mcp` in its `surfaces` (AEOS §4.9.4), the MCP server emits a tool. No per-tool manifest restatement of names or schemas.

2. **Naming.** The MCP tool name is the **`axiom_` server-prefix + the projector's surface name**:
   `capability "classroom.enroll"` → `capability_to_surface_name` → `classroom__enroll` → MCP `axiom_classroom__enroll`. This **preserves every existing MCP tool name** (the prior `axiom_{ext}__{verb}` convention is the same string when namespace == extension). Dispatch recovers the capability by stripping `axiom_` and applying `surface_to_capability_name`. The `axiom_` prefix is a transport concern (global tool-namespace uniqueness across MCP servers); the `.`⇄`__` mapping is the projector's.

3. **Schema.** The tool `inputSchema` is `inputs_to_json_schema(spec.inputs)` from the projector — the same schema the CLI and agent-tool use. The permissive `additionalProperties: True` default and the hand-crafted per-tool schemas are retired for registry-backed tools.

4. **Dispatch.** A registry-backed MCP tool dispatches through `SkillRegistry.invoke(capability_name, args, ctx)` — the same entry point the CLI verb uses — instead of a manifest-`entry` lazy import. One implementation, three surfaces.

5. **Side-effects / approval.** Read from the capability (`approval_category`), surfaced as the MCP `side_effects` annotation. Not restated in the manifest.

6. **Platform primitives stay special (bounded dual-source).** The seven always-on core tools (`axiom_memory__compose`, `axiom_rag__retrieve`, …) remain standalone handlers in `platform_primitives.py`. They are core platform services, not extension skills, and they sort first and cannot be shadowed (unchanged). A future ADR may register them as core capabilities; until then this is the *only* non-registry source, and it is fixed and enumerated — not a general escape hatch.

7. **`surfaces` gates exposure (the pollution guard).** A capability is an MCP tool **iff** it opts in via `surfaces = [... "mcp"]`. `SkillSpec` gains `surfaces` (additive, like `side_effects` in #511). This prevents "every registered skill becomes an MCP tool" — exposure stays declared and bounded (AEOS §4.9.4).

## Consequences

- One source of truth: a capability's MCP tool, CLI verb, and agent-tool now share name, schema, and approval semantics; the manifest-vs-registry drift ends.
- The independent mangler in `manifest_schema.py` and the per-tool schemas in `platform_primitives.py`/`aggregation.py` are replaced by `capability_projection` calls (the #22a mechanical consolidation), then the registry-driven discovery path is added (#22b).
- **Back-compat / migration (pre-launch, no deprecation aliases per `feedback_no_backward_compat_shims`):** during transition the aggregator accepts both — a capability is MCP-exposed if it declares `surfaces=["mcp", ...]` *or* a legacy `[[extension.mcp.tool]]` block names it; the legacy `entry`-handler path remains for any `kind=tool` that is not a registered skill. Once extensions declare `surfaces`, the `[[mcp.tool]]` blocks and `entry` lazy-import are retired.
- Existing MCP tool names are unchanged (the `axiom_` prefix + projector round-trip reproduce them), so MCP clients see no rename.
- The aggregation invariants (platform-first ordering, no-shadow, deterministic hash, opt-out) are preserved; new tests cover registry-derived names/schemas and `invoke`-based dispatch.
