# ADR-072 — Capability projection: one skill, many surfaces (CLI · MCP · agent-tool)

**Status:** Accepted · **Date:** 2026-06-09
**Owner:** @ben
**Builds on:** ADR-056 (skills as invocable functions), ADR-063 (SKILL.md as generated artifact), ADR-058 (Agent Standards Registry)
**Related:** ADR-006 / ADR-038 (MCP access + builtin MCP server), ADR-071 (`llm` core primitive), spec-aeos-0.1 §4.2/§4.3/§4.6/§4.9

## Context

ADR-056 made the registered function (`SkillSpec` in `axiom.infra.skills.SkillRegistry`) the **executable unit**: "the CLI verb invokes the skill; the agent persona invokes the *same* skill." ADR-063 went further — "`SkillRegistry` registration is the single source of truth" — and generates SKILL.md from it. That principle is right; it just hasn't been carried across *all* the surfaces a capability projects onto.

A capability is reachable on up to four surfaces:

| Surface | Source of truth today | Result |
|---|---|---|
| **CLI verb** (`axi ns verb`) | `SkillRegistry` (ADR-056) | ✅ aligned |
| **SKILL.md** (agentskills.io doc) | `SkillRegistry`, generated (ADR-063) | ✅ aligned |
| **MCP tool** | the **manifest** `[[extension.provides]] kind=tool` block (`prefix__tool`), *not* the registry | ❌ divergent source |
| **Agent-facing LLM tool** | new ad-hoc converter (`chat/skill_tools.py`, PR #474) | ❌ fourth convention |

Two structural problems follow:

1. **Two sources of truth.** CLI/SKILL.md derive from the registry; MCP derives from manifest `provides`-blocks. They can (and will) drift — a skill gains a parameter and only the CLI sees it.
2. **Triplicated machinery.** Three independent dotted→`__` name-manglers (`mcp/manifest_schema.py`, `chat/skill_tools.py`, `infra/hooks/registry.py`) and three independent `inputs → JSON Schema` generators (`mcp/platform_primitives.py`, `chat/tools.py`, `chat/skill_tools.py`). PR #474 adds a fourth name convention. Each is a place for drift and for tool-name collisions.

There is also a **terminology collision** that hides all this: the code's `SkillSpec` (an executable function with typed `inputs`) and AEOS §4.6 "skill" (a *SKILL.md* — natural-language instructions for an LLM) are different things wearing the same word.

And there is a **pollution hazard**: projecting "all skills" into every agent's LLM tool list is tool-explosion — dozens of `ns__verb` functions degrade tool selection and burn context. Nothing today *declares* which capabilities an agent may call.

## Decision

**The registered capability (`SkillSpec`) is the single source of truth. Its CLI verb, MCP tool, and agent-facing LLM tool are projections, all generated by one shared projector — `axiom.infra.capability_projection`. No surface and no extension defines its own name-mangling or schema translation.**

1. **One naming convention.** Dotted `ns.verb` ⇄ surface-safe `ns__verb` via a single shared round-trip (`capability_to_surface_name` / `surface_to_capability_name`). MCP tool names, agent-tool names, and SKILL.md names all derive from the capability name; manifests no longer restate them. MCP stops minting names from `[[provides]]` and derives them from the registry like everyone else.

2. **One schema derivation.** The JSON Schema is derived once from the capability's typed `inputs`/output. Every surface consumes that schema; none re-authors it. The hand-crafted per-tool schemas in `mcp/platform_primitives.py` become registry-derived.

3. **Side-effects & approval live on the capability, not the surface.** `SkillSpec` declares `side_effects` / `idempotent` (already required of `tool` in §4.2). READ/WRITE → approval-gating is read from that. Today the chat converter *defaults every skill to WRITE* — that policy belongs on the capability, declared once, honored by every surface (CLI confirm prompts, MCP `side_effects`, chat approval gate).

4. **Surface & agent exposure is declared and scoped.** A capability declares which surfaces it projects to (`surfaces = ["cli", "mcp", "agent_tool"]`) and which agents may call it. **An agent's LLM tool list is the scoped projection of the capabilities it is authorized to call — never "all skills."** This is the pollution fix: discovery stays dynamic, exposure stays bounded.

5. **Terminology is split** (normative, in the spec): the executable **capability/skill-function** (ADR-056) is distinct from the **SKILL.md** instruction document (agentskills.io). The spec language that conflates them is corrected.

These rules are written into **spec-aeos-0.1.md §4.9 (Capability projection)**, normative for every extension, and contradictory language in §4.2 (tool) and §4.6 (skill) is reconciled to point at it.

## Consequences

- One cohesive, dynamically-composed surface graph: add a skill once → it projects to CLI, MCP, and (scoped) agent tools with consistent names, schema, and approval semantics. No hand-maintained per-surface lists.
- The 3–4 name-manglers and 3 schema-generators collapse into `axiom.infra.capability_projection`; MCP/CLI/chat are migrated to consume it (incremental, pre-launch, no compat shims per `feedback_no_backward_compat_shims`).
- MCP tool discovery becomes registry-driven, ending the manifest-vs-registry drift; `[[provides]] kind=tool` blocks declare *exposure/metadata*, not duplicated names+schemas.
- Tool-explosion is structurally prevented: agents see a bounded, declared toolset.
- **PR #474** (`chat/skill_tools.py`) is reworked to sit *behind* the projector and read side-effects from the capability rather than defaulting WRITE — not merged as a fourth standalone convention.
- The agent-tool projection is the natural consumer of the ADR-071 `llm/` gateway (the gateway hands the model its scoped tool list).
- `SkillSpec` gains `side_effects`/`idempotent` and surface/exposure fields; SKILL.md generation (ADR-063) and the AEOS manifest extend to carry them. A CI lint guards single-source-of-truth (checked-in surfaces ≡ projector output), mirroring ADR-063's `--check`.
