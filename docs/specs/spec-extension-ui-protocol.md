# Axiom Extension UI Protocol — How Extensions Surface Capabilities to Users

**Status:** Draft
**Date:** 2026-04-16
**Owner:** Ben Booth
**Related:** `spec-classroom-generate-shortcuts.md` (Open WebUI adapter), prd-classroom.md, ADR-010 (CLI architecture), `project_extension_aware_rag_and_bonsai` memory, `feedback_axiom_extension_ui_protocol` memory.

---

## 1. Purpose

Axiom extensions add capabilities to the platform — new agents,
new data sources, new workflows, domain-specific knowledge. But
a capability that users can't discover doesn't exist. This spec
defines the protocol by which extensions **surface their
capabilities to users** across any frontend (Open WebUI, CLI,
Slack, Teams, future UIs) through a single registration
interface.

**Principle:** extensions register what they offer via Axiom's
protocol. UI adapters translate those registrations into
frontend-specific controls. Extensions never know which frontend
renders them. Adapters never know what domain the extension
serves.

---

## 2. Three Surface Types

Every user-visible extension capability maps to one of three
surface types:

### 2.1 Tiles — Pre-Action Discovery

**What:** clickable entry points shown BEFORE the user acts.
Tiles answer: "What can I do right now?"

**When visible:** empty state (new chat, fresh CLI session),
context transition (course phase change, role switch), idle
moments.

**Properties:**
- Static or slowly-changing (update per session, not per
  message).
- Contextual (70%) + tangential (30%). Contextual tiles
  match the user's current role/phase/workflow. Tangential
  tiles surface capabilities the user hasn't explored yet
  but that relate to what they've been doing.
- Each tile carries: icon, title, prompt template (with
  optional `$PLACEHOLDER`s), role gate, phase gate, priority.

**Tangential tile generation:** the system maintains a
per-user "capability coverage map" — which extension
capabilities the user has exercised. Tiles for un-exercised
capabilities that are topically related to the user's recent
activity get a `tangential: true` flag and a distinct visual
treatment ("You might also..." or "Discover:"). CURIO's
corpus-relatedness scoring determines topical relevance.

### 2.2 Actions — In-Context Operations

**What:** buttons attached to existing content (a message, a
search result, a document, a finding). Actions answer: "What
can I do with THIS?"

**When visible:** after content is rendered. Attached to the
content they operate on.

**Properties:**
- Deterministic trigger (button click), model-mediated
  response (the action's work may involve LLM).
- Context-specific: an action on a message with citations
  shows "Check citations"; an action on a grading queue item
  shows "Score with rubric." The action's `visible_when`
  condition determines this.
- Each action carries: id, icon, label, module (Python path
  to the handler), role gate, priority, `visible_when`
  condition.

### 2.3 Suggestions — Post-Action Next Steps

**What:** proposed next prompts shown AFTER the system responds.
Suggestions answer: "Where should I go next?"

**When visible:** after every system response, if the extension
has something relevant to suggest.

**Properties:**
- Model-mediated: generated by the extension's suggestion
  engine (typically LLM + RAG) based on the conversation
  context and the extension's domain knowledge.
- Mix of contextual ("dig deeper into [sub-topic from the
  response]") and tangential ("a classmate found a
  contradiction in a related area — investigate?").
- Each suggestion carries: text (the proposed prompt),
  source extension, confidence, contextual-vs-tangential
  flag.

## 3. Registration in `axiom-extension.toml`

Extensions declare their surfaces in the existing manifest:

```toml
[extension]
name = "classroom"
version = "0.1.0"
builtin = true

# --- UI SURFACES ---

[[ui.tiles]]
id = "research-topic"
icon = "🔬"
title = "Research a topic"
prompt_template = "/research $TOPIC"
placeholders = { TOPIC = "Enter a topic..." }
roles = ["student", "instructor"]
phases = ["active", "assessment"]
priority = 10
tangential = false

[[ui.tiles]]
id = "discover-peer-research"
icon = "🌐"
title = "See what classmates are researching"
prompt_template = "Show me what other students in this course are researching and where my work might connect."
roles = ["student"]
phases = ["active"]
priority = 90
tangential = true   # <-- non-linear discovery

[[ui.actions]]
id = "cite-check"
icon = "✓"
label = "Check citations"
module = "axiom.extensions.builtins.classroom.actions.cite_check"
roles = ["student", "instructor"]
priority = 10
visible_when = "message.has_citations or message.role == 'assistant'"

[[ui.actions]]
id = "submit-work"
icon = "📤"
label = "Submit as work"
module = "axiom.extensions.builtins.classroom.actions.submit"
roles = ["student"]
priority = 40
visible_when = "context.has_active_assignment"

[[ui.actions]]
id = "promote-finding"
icon = "⬆"
label = "Promote finding"
module = "axiom.extensions.builtins.classroom.actions.promote"
roles = ["instructor"]
priority = 10
visible_when = "message.has_finding"

[[ui.suggestions]]
id = "classroom-contextual"
module = "axiom.extensions.builtins.classroom.suggestions.contextual"
roles = ["student", "instructor"]
priority = 10

[[ui.suggestions]]
id = "classroom-tangential"
module = "axiom.extensions.builtins.classroom.suggestions.tangential"
roles = ["student"]
priority = 50
tangential = true
```

**Domain extensions add their own surfaces.** A domain consumer (e.g. an engineering assistant) adds:

```toml
[[ui.tiles]]
id = "search-domain-db"
icon = "🔬"
title = "Search domain reference databases"
prompt_template = "/search-domain reference-data $QUERY"
roles = ["student", "instructor"]
priority = 30

[[ui.actions]]
id = "validate-record"
icon = "📐"
label = "Validate against reference dataset"
module = "domain_consumer.extensions.actions.record_validate"
roles = ["student", "instructor"]
priority = 15
visible_when = "message.mentions_record"
```

These appear alongside classroom tiles/actions when both
extensions are configured. No code change in either extension.

## 4. Runtime Resolution

At runtime, the UI protocol resolves which surfaces to show:

```python
def resolve_surfaces(
    user: User,
    context: UIContext,
    extensions: list[Extension],
) -> ResolvedSurfaces:
    """Merge all registered surfaces, filter by role + phase + visibility,
    mix contextual and tangential, sort by priority."""
    
    tiles = []
    actions = []
    suggestions = []
    
    for ext in extensions:
        for tile in ext.ui_tiles:
            if user.role in tile.roles and context.phase in tile.phases:
                tiles.append(tile)
        for action in ext.ui_actions:
            if user.role in action.roles and action.visible_when(context):
                actions.append(action)
        for suggestion_provider in ext.ui_suggestions:
            if user.role in suggestion_provider.roles:
                suggestions.extend(
                    suggestion_provider.generate(context)
                )
    
    # Ensure tangential ratio
    contextual = [t for t in tiles if not t.tangential]
    tangential = [t for t in tiles if t.tangential]
    # Show at least 1 tangential per 3 contextual (target ~30%)
    merged_tiles = _interleave(contextual, tangential, ratio=0.3)
    
    return ResolvedSurfaces(
        tiles=sorted(merged_tiles, key=lambda t: t.priority),
        actions=sorted(actions, key=lambda a: a.priority),
        suggestions=suggestions,
    )
```

**The resolver is the protocol's core.** It's deterministic
(role/phase/visibility gates are code checks) with a
model-mediated tangential-scoring step (CURIO relevance).

## 5. UI Adapters

Each frontend implements an adapter that translates
`ResolvedSurfaces` into its native controls:

| Adapter | Tiles become | Actions become | Suggestions become |
|---------|-------------|---------------|-------------------|
| **OpenWebUIAdapter** | Model-level prompt suggestions (per-model config via admin API) | Action Functions (Python classes deployed to Open WebUI) | Structured "Next steps" section appended to response text → task model generates follow-up buttons |
| **CLIAdapter** | Argcomplete suggestions on empty input; `axi --suggest` command | `/slash` commands; inline `[action]` prompts after output | Printed "You might try:" section after each response |
| **GroupChatAdapter** | Vendor-specific buttons (Slack Block Kit / Mattermost attachments / Teams Adaptive Cards) in the app home or channel | Message action buttons in the platform's context menu | Thread/reply with suggestions |
| **IDEAdapter** | Editor command palette entries + sidebar panel tiles (VS Code/Cursor/Pear AI: TreeView items; JetBrains: tool window actions) | Inline code-lens actions on findings/citations in editor; right-click context menu entries | Inline suggestions after AI-assisted edits; "next step" entries in the Axiom sidebar panel |

GroupChat implementations: `SlackGroupChat`, `MattermostGroupChat`, `TeamsGroupChat`, others. Mattermost is self-hostable — relevant for air-gapped/classified deployments where Slack/Teams can't reach.

IDE implementations: `VSCodeIDE` (covers VS Code, Cursor, Pear AI — all VS Code forks sharing the extension API), `JetBrainsIDE` (IntelliJ platform — PyCharm, CLion, etc.), others as demand warrants. The VS Code family is the priority — Cursor and Pear AI are the AI-native forks students and researchers increasingly use. The adapter exposes Axiom surfaces via the editor's native extension points (command palette, TreeView sidebar, CodeLens, context menus) without requiring users to leave their editor.

**Adapter contract:**
- Accepts `ResolvedSurfaces`.
- Renders per its frontend's idiom.
- Returns analytics events (tile_clicked, action_clicked,
  suggestion_clicked) in a common schema regardless of frontend.
- Distinguishes tangential items visually (if the frontend
  supports it; if not, appends a "Discover:" label to the text).

**See `spec-classroom-generate-shortcuts.md`** for the Open WebUI
adapter's full implementation spec.

## 6. Tangential Discovery Engine

The tangential-discovery engine is the piece that makes shortcuts
more than a static menu. It runs per-user, periodically:

1. **Capability coverage map:** tracks which extension capabilities
   (tile IDs, action IDs) the user has exercised. "Student has
   used /research 12 times, /cite-check 3 times, /quiz-prep 0
   times, /discover-peer-research 0 times."

2. **Topical relevance scoring:** CURIO scores each un-exercised
   capability against the user's recent activity (last N
   interactions, current research topics, course objectives
   covered vs. uncovered).

3. **Tangential tile selection:** capabilities with high topical
   relevance AND low exercise count are promoted as tangential
   tiles. "You've been researching Topic A but haven't used
   /discover-peer-research, which would show you 3 classmates'
   related work."

4. **Decay:** tangential tiles that are repeatedly shown but
   never clicked decay in priority. The system respects that the
   user has seen and declined the suggestion.

5. **Analytics feedback:** which tangential tiles get clicked
   feeds back into the relevance model. Over time, the system
   learns which tangential suggestions are genuinely useful for
   which user profiles.

The tangential engine is **model-mediated** (CURIO + RAG
compute relevance scores). The tile visibility gate itself is
**deterministic** (must pass role + phase + extension-configured
constraints before the tangential engine even evaluates it).

## 7. Relation to Existing Extension Manifest

The `axiom-extension.toml` already defines:

```toml
[[cli.commands]]
noun = "research"
module = "axiom.extensions.builtins.classroom.research_cli"
```

The UI surfaces (`[[ui.tiles]]`, `[[ui.actions]]`,
`[[ui.suggestions]]`) are siblings of `[[cli.commands]]` in the
same manifest. A single extension registers both its CLI
interface and its UI surfaces in one file. Consistency: every
capability that has a CLI noun SHOULD have a corresponding tile
and/or action (if user-facing).

## 8. Test Strategy

- **Protocol tests:** verify `resolve_surfaces()` correctly
  filters by role, phase, visibility, and achieves the
  contextual/tangential ratio.
- **Per-adapter tests:** verify each adapter correctly translates
  `ResolvedSurfaces` to its frontend's native format.
- **Tangential engine tests:** verify coverage-map tracking,
  topical-relevance scoring, decay behavior, and analytics
  feedback loop.
- **Integration tests (per interaction axis):**
  - I↔S: instructor's "Score this response" action on a
    student's submission → Canvas grade push.
  - S↔S: "Discover peer research" tangential tile → shows
    cross-pollination results.
  - I↔I: "Share course template" tile → federation propagation.
